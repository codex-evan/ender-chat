# Security Architecture Document

## 1. Threat Model

### Assets to Protect
- Message content (text, images, files, voice)
- User identity (who is chatting)
- Room membership (which users are in which rooms)
- Local stored chat history
- Passphrase/recovery key

### Adversaries
1. **Server Operator**: Can see ciphertext, metadata, connection timing
2. **Network Attacker**: Can intercept network traffic
3. **Device Attacker**: Physical access to a device
4. **Platform OS**: OS-level screen capture (iOS/Android/Windows)
5. **Malware**: Running on user device

### Assumptions
- TLS 1.3 is properly configured for WSS connections
- Server hardware is trusted for processing ciphertext only
- Client devices may be compromised at varying degrees
- Users may share room codes with untrusted parties

### What We Protect Against
- Server-side message decryption
- Network eavesdropping on message content
- Historical data exposure after room ends
- Cross-room key compromise
- Replay attacks
- Message tampering
- Identity correlation

### Known Limitations
- Server sees connection metadata (timing, volume)
- Server knows room IDs exist (but not who created them after both leave)
- iOS cannot fully prevent screenshots (system limitation)
- Windows cannot fully prevent screen capture APIs
- Malware on device can capture displayed content

## 2. End-to-End Encryption

### Algorithm Selection
| Component | Algorithm | Rationale |
|-----------|-----------|-----------|
| Key Exchange | X25519 (Curve25519) | Modern, fast, secure ECDH |
| Message Encryption | AES-256-GCM / ChaCha20-Poly1305 | Authenticated encryption |
| Key Derivation | HKDF-SHA256 | RFC 5869 compliant KDF |
| Passphrase KDF | PBKDF2-SHA256 (100K iterations) | Widely supported, constant-time |
| Nonce | Random 96-bit (GCM) / 64-bit (Poly1305) | Per-message uniqueness |
| Message Signing | Ed25519 (optional) | Sender authenticity |

### Encryption Pipeline

```
Plaintext Message
       │
       ▼
┌──────────────┐
│ Add Metadata │  ← msg_id, type, timestamp
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ Encrypt      │  ← AES-256-GCM with per-message key + nonce
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ Package      │  ← {ciphertext, nonce, sender_ephemeral_pk, msg_id}
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ Sign (opt)   │  ← Ed25519 signature for authenticity
└──────┬───────┘
       │
       ▼
  Send to Server (all ciphertext)
```

## 3. Key Exchange Flow

### Double Ratchet Foundation

```
User A                          User B
  │                               │
  │  Identity Key Pair            │
  │  (pk_A, sk_A)                 │
  │                               │
  │  Ephemeral Key Pair           │
  │  (epk_A, esk_A)               │
  │                               │
  │───{pk_A, epk_A}─────────────>│
  │                               │
  │                               │  Identity Key Pair
  │                               │  (pk_B, sk_B)
  │                               │
  │                               │  Ephemeral Key Pair
  │                               │  (epk_B, esk_B)
  │                               │
  │<──{pk_B, epk_B}──────────────│
  │                               │
  │  Compute:                     │  Compute:
  │  ss1 = X25519(spk_A, pk_B)    │  ss2 = X25519(spk_B, pk_A)
  │  ss3 = X25519(epk_A, pk_B)    │  ss4 = X25519(epk_B, pk_A)
  │                               │
  │  Derive session keys:         │  Derive session keys:
  │  keys = HKDF(ss1||ss3, salt)  │  keys = HKDF(ss2||ss4, salt)
  │                               │
  │  Same keys derived ✓          │  Same keys derived ✓
```

### Room Salt
Each room gets a random 32-byte salt generated by the server during room creation. This salt ensures:
- Different rooms have different derived keys even with the same identity keys
- Past session keys cannot be derived from future ones

## 4. Room Code Generation

```
1. Generate 6 cryptographically random bytes (48 bits)
2. Map each byte to alphanumeric charset (36 chars)
3. Result: 6-character code with ~2^48 entropy
4. Server generates unique room UUID (128 bits)
5. Room code is a human-friendly representation
```

**Entropy Analysis:**
- 6 chars from 36-char alphabet = 2^48 ≈ 281 trillion combinations
- Brute-force resistance: ~10^14 attempts needed
- Rate limiting on server prevents automated attacks

## 5. Key Derivation from Passphrase

```
Passphrase + Salt (16 bytes)
       │
       ▼
┌──────────────┐
│ PBKDF2       │  ← SHA-256, 100,000 iterations
└──────┬───────┘
       │
       ▼
  256-bit Master Key
       │
       ├─→ Local Message Decryption Key
       ├─→ Local File Decryption Key
       └─→ Database Encryption Key
```

**Why PBKDF2 over Argon2id?**
- Argon2id requires native code bindings in Flutter
- PBKDF2 with 100K iterations provides adequate security
- Memory-hard algorithms not critical for local storage

## 6. File Encryption

### Chunked Encryption Strategy
```
File (up to 100MB)
       │
       ▼
┌──────────────┐
│ Split into   │  ← 1MB chunks
│ 1MB chunks   │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ Encrypt each │  ← Independent nonce per chunk
│ chunk        │  ← AES-256-GCM with file key
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ Upload       │  ← Server stores encrypted chunks
│ chunks       │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ Download &   │  ← Decrypt each chunk independently
│ decrypt      │
└──────────────┘
```

### Filename Encryption
Original filenames are encrypted as part of the file metadata blob, which is itself encrypted with the file key. The server only sees:
- Encrypted file ID (UUID)
- Encrypted blob size
- Number of chunks
- Encrypted metadata

## 7. Anti-Screenshot/Screen-Recording

### Android (Best Protection)
```
FLAG_SECURE set on activity
       │
       ├─→ Screenshots blocked (black frame)
       ├─→ Screen recording blocked (black frame)
       ├─→ Recent tasks shows black thumbnail
       └─→ ADB screenshot blocked
```

### iOS (Detection Only)
```
UIApplication.userDidTakeScreenshotNotification
       │
       ├─→ Detect screenshot event
       ├─→ Blur chat interface immediately
       ├─→ Notify other participant
       └─→ Log security event
```

**Limitation**: iOS does not allow preventing screenshots at the system level. Detection is the best available approach.

### Windows (Best-Effort)
```
DWM APIs + Window attributes
       │
       ├─→ Blur on background switch
       ├─→ DWM_DISK_GPU_FILTER discourage capture
       ├─→ Foreground change detection
       └─→ Warning dialog if suspicious activity
```

**Limitation**: Windows has no built-in screen capture prevention API for arbitrary applications.

### Protection Matrix

| Platform | Prevent Screenshot | Prevent Recording | Detect Capture | Blur UI |
|----------|-------------------|-------------------|----------------|---------|
| Android  | ✅ Full           | ✅ Full           | Partial        | ✅ Yes  |
| iOS      | ❌ No (system)    | ❌ No (system)    | ✅ Yes          | ✅ Yes  |
| Windows  | ❌ No (system)    | ❌ No (system)    | Limited        | ✅ Yes  |

## 8. Server Data Minimization

### What Server Stores
- Room UUID (hashed for lookup)
- Ciphertext (never decrypted)
- Nonce (for message deduplication)
- Timestamps (for TTL expiry)
- Encrypted file blobs (never inspected)

### What Server Does NOT Store
- Plaintext messages
- Plaintext files
- Filenames (encrypted)
- File contents
- Keys (any kind)
- Passphrases
- User identities
- Phone numbers / emails
- IP addresses in logs

### Log Sanitization
All server logs strip or redact:
- Message ciphertext
- Nonces
- Keys
- Room codes (only hash logged)
- File names
- Any field matching sensitive patterns

## 9. Data Deletion Strategy

### Immediate Deletion
- Both participants leave room → server destroys room data
- Room destroy requested → all data purged
- User deletes local record → local data wiped

### TTL-Based Deletion
- Messages: 7 days from creation (configurable)
- Files: 7 days from upload completion
- Rate limit entries: 60 seconds sliding window
- Cleanup runs every 60 minutes

### Deletion Verification
After deletion, server verifies:
- Database records removed
- File blobs deleted from storage
- In-memory state cleared
- No backup retention (or encrypted backups only)

## 10. Anti-Replay & Integrity

### Message Integrity
```
Each message includes:
├── msg_id (unique UUID)
├── timestamp (epoch ms)
├── type (enum)
├── ciphertext (encrypted payload)
└── signature (Ed25519, optional)
```

### Replay Prevention
1. **msg_id deduplication**: Server rejects duplicate message IDs within a room
2. **Timestamp window**: Messages with timestamps > 5 minutes in the future are rejected
3. **Nonce uniqueness**: Each message uses a unique nonce; reuse detected via storage lookup
4. **Sequence numbers**: Optional monotonic counter for message ordering

### Tamper Detection
- AES-GCM authenticated encryption detects any modification
- Ed25519 signatures verify sender authenticity (when enabled)
- Server cannot modify ciphertext without detection

## 11. Rate Limiting & DoS Prevention

### Per-Client Limits
| Action | Limit | Window |
|--------|-------|--------|
| WebSocket messages | 100 | 60 seconds |
| Room joins | 10 | 60 seconds |
| File uploads | 5 | 60 minutes |
| Message size | 1 MB | - |
| File size | 100 MB | - |

### Room Limits
- Maximum 2 participants per room (design constraint)
- Room lifetime: auto-destroy when both leave
- Concurrent rooms: unlimited per client

### Server Hardening
- Request body size limits enforced
- WebSocket max payload: 100 MB
- Connection timeout: 30 seconds idle
- Graceful degradation under load

## 12. Cryptographic Summary

| Property | Implementation |
|----------|---------------|
| Forward secrecy | ✅ Ephemeral keys per session |
| Post-compromise security | ✅ Key rotation on rejoin |
| Authentication | ✅ Ed25519 signatures (optional) |
| Integrity | ✅ AES-GCM authenticated encryption |
| Replay protection | ✅ msg_id dedup + timestamp window |
| Key separation | ✅ Separate keys for messages, files, auth |
| Random nonce | ✅ CSPRNG per message |
| No hardcoded keys | ✅ All keys derived dynamically |
